# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Auto-sell is a multi-tenant SaaS platform that automates WhatsApp-based sales for small businesses. Business owners upload a product catalog; an LLM-powered bot (DeepSeek) handles customer conversations, shares product media on demand, negotiates within configured price ranges, generates payment links (Paystack), and alerts the business owner when a sale is confirmed.

## Stack

- **Django 6 + Django Ninja** — Django Admin for platform ops, Ninja for the REST API (`/api/docs` for auto-generated docs)
- **PostgreSQL 16 + pgvector image** — primary DB; `SearchVectorField` (GIN index) for full-text catalog search
- **Redis 7** — Celery broker + live conversation history store
- **Celery 5** — async processing: LLM calls, WhatsApp sends, payment links, owner notifications
- **DeepSeek Chat API** — via `openai` SDK (`base_url="https://api.deepseek.com/v1"`, `model="deepseek-chat"`)
- **WhatsApp Business Cloud API** — webhook-driven inbound, Graph API for outbound
- **Paystack** — payment link generation; gateway interface in `apps/payments/gateways/base.py` is pluggable
- **Cloudflare R2** — media storage via boto3 (S3-compatible)

## Development

```bash
# Start infrastructure
docker compose up -d

# Install deps (first time)
uv venv .venv && source .venv/bin/activate
uv pip install -r requirements.txt   # or: uv sync

# Database
python manage.py migrate
python manage.py createsuperuser

# Run (4 terminals)
python manage.py runserver
celery -A auto_sell worker -l info --pool=solo   # --pool=solo required on macOS
celery -A auto_sell beat -l info                 # periodic tasks (abandoned sweep, etc.)
ngrok http 8000                                   # expose webhook to Meta
```

Run tests: `pytest tests/ apps/ -v`

API docs: `http://localhost:8000/api/docs`

## App Structure

- `apps/tenants/` — `Tenant` and `TenantUser`. Platform admins onboard businesses via Django Admin. WhatsApp credentials and owner alert contact live on `Tenant`. `Tenant.slug` is used in the webhook URL path.
- `apps/catalog/` — `Product` and `ProductMedia`. `price_min`/`price_max` define the range the LLM is allowed to quote. `search_vector` is auto-updated by a `post_save` signal. `ProductMedia.wa_media_id` caches the WhatsApp media ID after first upload to avoid re-uploading.
- `apps/conversations/` — Core of the service. `Conversation` tracks state (`active → awaiting_payment → completed/abandoned/escalated`). Live history is a Redis List at `conversation:{uuid}:history` (TTL 72h, max 20 entries). A Redis lock at `conversation:{uuid}:lock` (TTL 30s) prevents duplicate processing on Meta retries. Product context is cached at `conversation:{uuid}:products`. A per-customer rate limit is enforced via a Redis counter at `rate:{tenant_id}:{customer_wa_id}` (max 10 messages per 60s fixed window); breaching it sends a "slow down" reply and exits before the lock or LLM. `WhatsAppClient` in `whatsapp.py` handles all outbound API calls. The Celery task `process_message` runs the FTS search, assembles the system prompt, calls DeepSeek, parses tool calls, and dispatches replies. Five edge cases are explicitly handled: (1) a returning customer on a `completed`/`abandoned` conversation has their state reset to `active` and Redis cache cleared; (2) a message during `awaiting_payment` checks the latest `PaymentLink` — if still `pending` it sends a reminder, if `expired`/`failed`/absent it resets to `active` so the LLM can re-engage; (3) non-text inbound messages (images, voice, etc.) dispatch `reply_unsupported_message` instead of being silently dropped (also rate-limited; silently dropped when over limit); (4) a message on an `escalated` conversation is silently dropped — the bot stays completely silent until the owner manually resets state to `active` via Django Admin; (5) `sweep_abandoned_conversations` is a Celery Beat task (runs hourly) that bulk-marks `active` conversations abandoned after 24h of inactivity and `awaiting_payment` conversations after 48h. When the LLM calls `escalate_to_human`, the dispatch sets state to `escalated` and enqueues `notify_owner_escalation` (in `apps/notifications/tasks.py`) which sends a WhatsApp alert to `tenant.owner_phone` with the customer ID and reason. The Paystack webhook also handles `charge.failed` — marking the link as failed and resetting the conversation to `active` proactively. `process_message` retries up to 3 times on transient failures (30s delay). `alert_owner` retries up to 5 times (60s delay) and logs to `NotificationLog` only on final failure.
- `apps/payments/` — `PaymentLink` and `Sale`. Gateway ABC in `gateways/base.py`; Paystack is the concrete implementation. Payment links are generated by the LLM calling the `generate_payment_link` tool. `Sale` is created when the Paystack `charge.success` webhook is received.
- `apps/notifications/` — `NotificationLog` + Celery tasks that send a WhatsApp alert to `tenant.owner_phone` when a `Sale` is created.

## LLM Conversation Design

Each `process_message` Celery task:
1. Resets conversation state to `active` if it was `completed` or `abandoned` (returning customer)
2. If state is `awaiting_payment`, checks the most recent `PaymentLink` status: if still `pending`, sends a reminder and returns; if `expired`, `failed`, or absent, resets to `active` and falls through to the LLM so the customer can re-negotiate
3. Full-text searches the tenant's catalog for terms in the inbound message (top 5 products)
4. Assembles a 3-part system prompt: (a) identity + rules, (b) FTS results as JSON, (c) tool definitions
5. Calls DeepSeek with tools: `send_product_media`, `generate_payment_link`, `escalate_to_human`
6. Parses `tool_calls` and executes them

The LLM enforces price range constraints. Never hardcode prices or bypass the `price_min`/`price_max` check.

## WhatsApp Webhook

Inbound: `POST /api/webhooks/whatsapp/{tenant_slug}/`
- Verify `X-Hub-Signature-256` against `tenant.wa_app_secret`
- Payload validated against `WAWebhookPayload` Pydantic schema (`apps/conversations/schemas.py`) — malformed payloads are logged and return 200 without processing
- Deduplicate via `Message.wa_message_id` before enqueueing
- Non-text messages (images, audio, stickers) dispatch `reply_unsupported_message` — never silently dropped
- Must return HTTP 200 in under 5 seconds — all processing is async in Celery

Outbound: `WhatsAppClient` calls `https://graph.facebook.com/v19.0/{phone_number_id}/messages`
