## Project

Auto-sell is a multi-tenant SaaS platform that automates WhatsApp-based sales for small businesses. Business owners upload a product catalog; an LLM-powered bot (DeepSeek) handles customer conversations, shares product media on demand, negotiates within configured price ranges, generates payment links (Paystack), and alerts the business owner when a sale is confirmed.

## Stack

- **Django 5 + Django Ninja** ŌĆö Django Admin for platform ops, Ninja for the REST API (`/api/docs` for auto-generated docs)
- **PostgreSQL 16 + pgvector image** ŌĆö primary DB; `SearchVectorField` (GIN index) for full-text catalog search
- **Redis 7** ŌĆö Celery broker + live conversation history store
- **Celery 5** ŌĆö async processing: LLM calls, WhatsApp sends, payment links, owner notifications
- **DeepSeek Chat API** ŌĆö via `openai` SDK (`base_url="https://api.deepseek.com/v1"`, `model="deepseek-chat"`)
- **WhatsApp Business Cloud API** ŌĆö webhook-driven inbound, Graph API for outbound
- **Paystack** ŌĆö payment link generation; gateway interface in `apps/payments/gateways/base.py` is pluggable
- **Cloudflare R2** ŌĆö media storage via boto3 (S3-compatible)

## Development

```bash
# Start infrastructure
docker compose up -d

# Install deps (first time)
uv venv .venv && source .venv/bin/activate
uv pip install -r requirements.txt   # or: uv sync

# Database
python manage.py migrate
python manage.py createsuperuser

# Run (3 terminals)
python manage.py runserver
celery -A auto_sell worker -l info --pool=solo   # --pool=solo required on macOS
ngrok http 8000                                   # expose webhook to Meta
```

Run tests: `pytest apps/ -v`

API docs: `http://localhost:8000/api/docs`

## App Structure

- `apps/tenants/` ŌĆö `Tenant` and `TenantUser`. Platform admins onboard businesses via Django Admin. WhatsApp credentials and owner alert contact live on `Tenant`. `Tenant.slug` is used in the webhook URL path.
- `apps/catalog/` ŌĆö `Product` and `ProductMedia`. `price_min`/`price_max` define the range the LLM is allowed to quote. `search_vector` is auto-updated by a `post_save` signal. `ProductMedia.wa_media_id` caches the WhatsApp media ID after first upload to avoid re-uploading.
- `apps/conversations/` ŌĆö Core of the service. `Conversation` tracks state (`active ŌåÆ awaiting_payment ŌåÆ completed/abandoned`). Live history is a Redis List at `conversation:{uuid}:history` (TTL 72h, max 20 entries). A Redis lock at `conversation:{uuid}:lock` (TTL 30s) prevents duplicate processing on Meta retries. `WhatsAppClient` in `whatsapp.py` handles all outbound API calls. The Celery task `process_message` runs the FTS search, assembles the system prompt, calls DeepSeek, parses tool calls, and dispatches replies.
- `apps/payments/` ŌĆö `PaymentLink` and `Sale`. Gateway ABC in `gateways/base.py`; Paystack is the concrete implementation. Payment links are generated by the LLM calling the `generate_payment_link` tool. `Sale` is created when the Paystack `charge.success` webhook is received.
- `apps/notifications/` ŌĆö `NotificationLog` + Celery tasks that send a WhatsApp alert to `tenant.owner_phone` when a `Sale` is created.

## LLM Conversation Design

Each `process_message` Celery task:
1. Full-text searches the tenant's catalog for terms in the inbound message (top 5 products)
2. Assembles a 3-part system prompt: (a) identity + rules, (b) FTS results as JSON, (c) tool definitions
3. Calls DeepSeek with tools: `send_product_media`, `generate_payment_link`, `escalate_to_human`
4. Parses `tool_calls` and executes them

The LLM enforces price range constraints. Never hardcode prices or bypass the `price_min`/`price_max` check.

## WhatsApp Webhook

Inbound: `POST /api/webhooks/whatsapp/{tenant_slug}/`
- Verify `X-Hub-Signature-256` against `tenant.wa_webhook_verify_token`
- Deduplicate via `Message.wa_message_id` before enqueueing
- Must return HTTP 200 in under 5 seconds ŌĆö all processing is async in Celery

Outbound: `WhatsAppClient` calls `https://graph.facebook.com/v19.0/{phone_number_id}/messages`

## Settings

`DJANGO_SETTINGS_MODULE` defaults to `auto_sell.settings.local` (set in `manage.py` and `celery.py`). Production uses `auto_sell.settings.production`. All secrets come from `.env` via `django-environ`.

Key env vars: `DATABASE_URL`, `REDIS_URL`, `DEEPSEEK_API_KEY`, `PAYSTACK_SECRET_KEY`, `S3_ENDPOINT_URL`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_BUCKET_NAME`. See `.env.example`.
